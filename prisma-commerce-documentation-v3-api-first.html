Prisma Commerce v1 Technical Specification

A commerce-first Laravel platform with CMS, themes, plugins, headless APIs, mobile app support, and third-party integrations.

Laravel + FilamentAPI FirstHeadless ReadyPlugin FriendlyTheme Friendly

1. Project Overview

Prisma Commerce is a Laravel-based eCommerce platform designed to help businesses build online stores, manage products and orders, publish content, install themes, install plugins, and connect with mobile apps or third-party websites through APIs.

Primary goal: build a Shopify + WooCommerce + Botble-inspired commerce platform, but with an independent architecture and API-first design.

2. Core Principles

Commerce First

Products, customers, carts, checkout, orders, payments, shipping, coupons, and inventory are first-class modules.

API First

Everything useful to storefronts, mobile apps, Next.js websites, integrations, and AI agents must be available through versioned APIs.

Developer Friendly

Clear modules, documented APIs, stable hooks, plugin manifests, theme manifests, and automated tests.

Theme Friendly

Themes control storefront presentation without modifying core code or database structure.

Plugin Friendly

Plugins extend features through events, hooks, routes, migrations, settings, permissions, and admin menus.

Community Ready

The platform should support a future plugin/theme marketplace and migration guides for other ecosystems.

17. API-First Strategy

Prisma Commerce must support traditional themes and fully headless usage. A business should be able to use Prisma Commerce as the backend for a Blade theme, Next.js storefront, Nuxt storefront, Flutter app, React Native app, or external marketplace integration.

Requirement: every customer-facing storefront feature must be available through API v1.

API Groups

/api/v1/store/*        Public storefront/headless API
/api/v1/customer/*     Authenticated customer account API
/api/v1/content/*      CMS/blog/pages/menu/SEO API
/api/v1/admin/*        Admin/system API for dashboards and internal tools
/api/v1/integration/*  Third-party integration API
/api/v1/webhooks/*     Webhook management and delivery logs

18. API Standards

Versioning

All endpoints start with /api/v1. Never break v1 responses after public release.

Authentication

Public endpoints are open. Customer endpoints use Sanctum. Admin/integration endpoints use permissions and API keys.

Response Format

Use consistent JSON: success, data, message, errors, meta.

Pagination

List endpoints support page, per_page, sort, filter, and include.

Rate Limiting

Apply different limits for public, customer, admin, and integration APIs.

CORS

Configurable allowed origins for Next.js, mobile apps, and third-party frontends.

Standard Response

{
  "success": true,
  "data": {},
  "message": null,
  "errors": null,
  "meta": {
    "version": "v1"
  }
}

25. Mobile App & Next.js Headless Support

Next.js Storefront Requirements

  • All product, category, brand, page, blog, menu, SEO, cart, checkout, and order confirmation data must be available through APIs.
  • CORS must allow configured storefront domains.
  • SEO metadata must be available by route or entity.
  • Product filters must be API-readable.
  • Cart must work without Laravel sessions, using cart IDs or tokens.
  • Checkout must work through JSON APIs.

Mobile App Requirements

  • Token-based customer authentication.
  • Device token registration for push notifications.
  • Customer profile, addresses, wishlist, orders, returns, downloads, reviews.
  • API-friendly image URLs and thumbnails.
  • Consistent error responses.
  • Pagination for all list endpoints.

Headless Modes

Mode 1: Prisma Commerce + Blade Theme
Mode 2: Prisma Commerce + Next.js Storefront
Mode 3: Prisma Commerce + Flutter / React Native App
Mode 4: Prisma Commerce + External Marketplace / ERP
Mode 5: Prisma Commerce as API-only backend

Prisma Commerce should cover these categories and extend them with admin, integration, and webhook APIs.

27. Testing Requirements

  • Every module must include unit and feature tests.
  • Every API endpoint must include success, validation, permission, and unauthenticated tests.
  • Every permission must be tested.
  • Plugin installation, activation, deactivation, and removal must be tested.
  • Theme activation must be tested.
  • Minimum target coverage: 80%.

28. Security Requirements

  • Never expose admin APIs without authentication and permissions.
  • Use signed webhooks with retry and delivery logs.
  • Use rate limiting for public endpoints.
  • Use CORS allowlist, not unrestricted production CORS.
  • Validate uploaded theme and plugin packages.
  • Do not allow themes to execute migrations.
  • Do not store plaintext API keys after display; store hashed keys.
  • Audit log admin actions and plugin/theme installation.

30. AI Agent Build Rules

If an AI coding agent is used, it must follow this document as the source of truth.

  • Use staging server only.
  • Use feature branches, not direct commits to main.
  • Build milestones in order.
  • Write tests for every module and API endpoint.
  • Document every new API route.
  • Do not add packages without explaining why.
  • Do not edit vendor files.
  • Do not hardcode settings or permissions.
  • Do not introduce breaking API response changes.
